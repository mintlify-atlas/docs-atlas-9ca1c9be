D&D Textil is a full-stack SaaS application built on a classic three-tier architecture: a React 18 frontend, a Java HTTP server backend, and a PostgreSQL database.

## System tiers

| Tier | Technology | Port |
|------|-----------|------|
| Frontend | React 18 + Vite + Tailwind CSS | `3001` (dev) |
| Backend | Java HTTP server (`com.sun.net.httpserver`) | `8081` |
| Database | PostgreSQL | `5432` |

## Request flow

Every user interaction follows this path:

```
Browser (React)
  вҶ’ fetch('http://localhost:8081/api/...')
  вҶ’ Java ApiServer (HttpServer)
  вҶ’ DAO layer (JDBC PreparedStatement)
  вҶ’ PostgreSQL (tienda_digital_textiles_db)
  вҶҗ JSON response
  вҶҗ React Context state update
  вҶҗ UI re-render
```

<Note>
The Vite dev server proxies `/uploads` requests to `http://localhost:8081`, so product images served from the Java backend are accessible at the same origin as the frontend during development.
</Note>

## Key design decisions

<CardGroup cols={2}>
  <Card title="Context API for state" icon="layers">
    Six React Context providers (Auth, Cart, Product, Notification, UI, Metrics) wrap the entire app in `AppProviders.jsx`. No external state library (Redux/Zustand) is used.
  </Card>
  <Card title="localStorage persistence" icon="database">
    The `useLocalStorage` custom hook persists cart items, authenticated user, dark mode preference, and applied coupon across browser sessions.
  </Card>
  <Card title="SHA-256 passwords" icon="lock">
    `AuthDAO.java` hashes passwords with `MessageDigest.getInstance("SHA-256")` before storing them. The `password_hash` field is stripped from the user object before it reaches `localStorage`.
  </Card>
  <Card title="Graceful offline fallback" icon="wifi-off">
    `ProductContext` and `MetricsContext` ship with static mock data. If the Java backend is unreachable, the app continues to function with the bundled fallback data.
  </Card>
</CardGroup>
